# transactions — domain module

Rules: ../../../docs/architecture/archetypes/domain-module.md

Owns the Transaction rows and the per-connection sync cursors; hosts the recurring bank-sync task. Domain authority: [`docs/domain/README.md`](../../../docs/domain/README.md) §Transactions and §Sync orchestration.

Module-specific notes:
- Provider-agnostic: reaches the bank only through the injected `banking.BankProvider` seam and `banking.ErrReauthRequired`. It must never import `plaid` (enforced by `architecture/isolation_test.go`).
- **One-way dependencies:** `transactions` imports `accounts.Service` and `categorization.Service`; neither calls back into Transactions. `SyncTransactions` calls `accounts.SyncAccounts` FIRST (balances + connection health) before writing any transaction rows, then pulls per connection — keeping the module graph an acyclic DAG.
- **Categorization (this module is the only writer).** After all connections' pulls, `SyncTransactions` runs a **self-healing sweep** (`categorizeUncategorizedSweep`): it resolves every still-uncategorized (`classification = ''`), non-overridden row across the **whole stored set**, not just this pull's delta — the categorization analogue of the transfer-pairing re-scan. So a row a prior sync left uncategorized (synced before categorization ran, or after a categorize error that still advanced the cursor) resolves on the **next** sync instead of staying stuck until a full DB re-backfill. `syncConnection` therefore does **not** categorize — keep the cursor advance and categorization decoupled so a categorize failure never orphans rows behind an advanced cursor. The bank-sync upsert deliberately never touches `classification`/`category_id`, so an existing row's categorization is preserved; once resolved (any non-empty classification) a row is out of the sweep's scope. `ReCategorize` records a manual override (`categorization_overridden = 1`, sticky across re-sync; a Spending choice requires a Category, Income/Transfer/needs-review clear it). `ApplyCategorization(substrings)` re-resolves matching non-overridden rows and is driven by the server-wired re-categorize seam on a Rule change. Categorization resolution runs on the global handle after the upsert transactions commit.
- It gets the connections to pull from `accounts.ConnectionsToSync`, which carries each connection's decrypted access token and its provider→internal account-id map. Decryption happens inside `accounts` (the field's owner); transactions resolves each pulled row's account via that map.
- `repo.go` is the only file that touches `core/db/sqlc`; its methods take and return this package's domain types (`Transaction`, `RecentTransaction`), never `sqlc.*`.
- Rows are keyed by the **stable provider transaction id** (PK). `added`/`modified` upsert in place; `removed` ids delete by id — so a re-sync over unchanged data is idempotent (no duplicates). Amounts are stored signed as-is from the seam (outflow positive, inflow negative).
- Per-connection failure isolation: a pull that returns `banking.ErrReauthRequired` is skipped (its cursor left unchanged) and the pass continues; the overall sync does not error. Mirrors `accounts.syncConnection`.
- Each connection's resume cursor lives in `transaction_sync_state` (PK `connection_id`). A fresh connection starts from the empty cursor (full backfill) and the returned cursor is persisted in the same transaction as the row writes, so a partial apply never leaves the cursor ahead of the data.
- The bank's two-level category strings are still stored verbatim (`category_primary` / `category_detailed`) as the input to resolution; the resolved internal facet lives in the separate `classification` / `category_id` / `categorization_overridden` columns.
- **Transfer subtype/destination pairing (shipped, ADR-0003 layer 2).** After categorization, `resolveTransferDestinations` re-resolves each non-overridden outflow Transfer leg's destination + subtype against every stored leg (the re-resolve-from-scratch rationale is in its code comment; the matcher is `categorization.ResolveTransferSubtype`). `MarkTransferDestination` records a manual override (`transfer_destination_overridden = 1`). Like the categorization facet, the transfer facet columns are **deliberately omitted from the bank-sync `UpsertTransaction` INSERT/ON-CONFLICT** so a re-sync never clobbers a user override — **never add the facet columns to the upsert.**
- The same stored-leg re-pairing is exposed as a public method (no provider call) so an `accounts` kind/savings override can apply immediately: it is wired at the composition root into the accounts override seam ([ADR-0008](../../../docs/adr/0008-account-kind-and-savings-overrides.md)), mirroring how a Rule change drives `ApplyCategorization`. Re-pairing skips manually-overridden Transfers either way.
- **`/transactions` view filters query full history; the resolve handlers are view-aware.** Merchant search (cleaned merchant) and the [needs-attention](../../../docs/domain/README.md) filter (`?view=needs-attention`) both read the full table, not just the default view's recent cap. In the needs-attention view, `ReCategorize`/`MarkTransferDestination` must drop a row that no longer qualifies (the worklist shrinks); in the default view the same edit re-renders the row in place — do not collapse the two render paths into one.
- The recurring sync lives in `task.go` (implements `core/task.Task`) and is registered at the composition root (`server/services.go`).
