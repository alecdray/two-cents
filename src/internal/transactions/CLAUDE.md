# transactions — domain module

Rules: ../../../docs/architecture/archetypes/domain-module.md

Owns the Transaction rows and the per-connection sync cursors; hosts the recurring bank-sync task. Domain authority: [`docs/domain/README.md`](../../../docs/domain/README.md) §Transactions and §Sync orchestration.

Module-specific notes:
- Provider-agnostic: reaches the bank only through the injected `banking.BankProvider` seam and `banking.ErrReauthRequired`. It must never import `plaid` (enforced by `architecture/isolation_test.go`).
- **One-way dependencies:** `transactions` imports `accounts.Service` and `categorization.Service`; neither calls back into Transactions. `SyncTransactions` calls `accounts.SyncAccounts` FIRST (balances + connection health) before writing any transaction rows, then pulls per connection — keeping the module graph an acyclic DAG.
- **Categorization (this module is the only writer).** After all connections' pulls, `SyncTransactions` runs a **self-healing sweep** (`categorizeUncategorizedSweep`): it resolves every still-uncategorized (`classification = ''`), non-overridden row across the **whole stored set**, not just this pull's delta — the categorization analogue of the transfer-pairing re-scan. So a row a prior sync left uncategorized (synced before categorization ran, or after a categorize error that still advanced the cursor) resolves on the **next** sync instead of staying stuck until a full DB re-backfill. `syncConnection` therefore does **not** categorize — keep the cursor advance and categorization decoupled so a categorize failure never orphans rows behind an advanced cursor. The bank-sync upsert deliberately never touches `classification`/`category_id`, so an existing row's categorization is preserved; once resolved (any non-empty classification) a row is out of the sweep's scope. `ReCategorize` records a manual override (`categorization_overridden = 1`, sticky across re-sync; a Spending choice requires a Category, Income/Transfer/needs-review clear it; moving **off** Transfer also clears the transfer destination + subtype, since Reporting counts a Savings contribution by subtype alone and a stale subtype would double-count — see the domain ReCategorize card). `ApplyCategorization(substrings)` re-resolves matching non-overridden rows and is driven by the server-wired re-categorize seam on a Rule change. Categorization resolution runs on the global handle after the upsert transactions commit.
- It gets the connections to pull from `accounts.ConnectionsToSync`, which carries each connection's decrypted access token and its provider→internal account-id map. Decryption happens inside `accounts` (the field's owner); transactions resolves each pulled row's account via that map.
- `repo.go` is the only file that touches `core/db/sqlc`; its methods take and return this package's domain types (`Transaction`, `RecentTransaction`), never `sqlc.*`.
- Rows are keyed by the **stable provider transaction id** (PK). `added`/`modified` upsert in place; `removed` ids delete by id — so a re-sync over unchanged data is idempotent (no duplicates). Amounts are stored signed as-is from the seam (outflow positive, inflow negative).
- Per-connection failure isolation: a pull that returns `banking.ErrReauthRequired` is skipped (its cursor left unchanged) and the pass continues; the overall sync does not error. Mirrors `accounts.syncConnection`.
- Each connection's resume cursor lives in `transaction_sync_state` (PK `connection_id`). A fresh connection starts from the empty cursor (full backfill) and the returned cursor is persisted in the same transaction as the row writes, so a partial apply never leaves the cursor ahead of the data.
- The bank's two-level category strings are still stored verbatim (`category_primary` / `category_detailed`) as the input to resolution; the resolved internal facet lives in the separate `classification` / `category_id` / `categorization_overridden` columns.
- **Bank display-detail columns ARE in the upsert** ([ADR-0013](../../../docs/adr/0013-richer-bank-transaction-detail.md)). The read-only display detail (raw descriptor, merchant logo/website/entity id, payment channel, categorization confidence, authorized/posted timestamps, and the counterparties JSON) is bank-sourced and carries no user state, so `UpsertTransaction` writes it and every sync refreshes it — the exact opposite of the override-facet columns above. Keep the two groups straight: facet columns stay out of the upsert; display-detail columns stay in. The counterparties list is a JSON column (display-only — never queried or matched; categorization still resolves on the cleaned merchant). Existing rows backfill by clearing the `transaction_sync_state` cursor so the next sync re-pulls and re-upserts.
- **Transfer subtype/destination pairing (shipped, ADR-0003 layer 2).** After categorization, `resolveTransferDestinations` re-resolves each non-overridden outflow Transfer leg's destination + subtype against every stored leg (the re-resolve-from-scratch rationale is in its code comment; the matcher is `categorization.ResolveTransferSubtype`). `MarkTransferDestination` records a manual override (`transfer_destination_overridden = 1`). Like the categorization facet, the transfer facet columns are **deliberately omitted from the bank-sync `UpsertTransaction` INSERT/ON-CONFLICT** so a re-sync never clobbers a user override — **never add the facet columns to the upsert.**
- The same stored-leg re-pairing is exposed as a public method (no provider call) so an `accounts` kind/savings override can apply immediately: it is wired at the composition root into the accounts override seam ([ADR-0008](../../../docs/adr/0008-account-kind-and-savings-overrides.md)), mirroring how a Rule change drives `ApplyCategorization`. Re-pairing skips manually-overridden Transfers either way.
- **`/transactions` view filters query full history.** Merchant search (cleaned merchant) and the [needs-attention](../../../docs/domain/README.md) filter (`?view=needs-attention`) both read the full table, not just the default view's recent cap.
- **Account display names are resolved through `accounts`, not joined.** The `RecentTransaction` read model carries the source + destination account **ids**; the service fills their display names via `accounts.DisplayNames` (the custom-name precedence lives in `accounts`, [ADR-0017](../../../docs/adr/0017-custom-account-names.md)). The queries no longer select `accounts.name`. They still join `accounts` for `mask` and `categories` for the category name — a residual raw cross-module read tracked in [known-gaps](../../../docs/architecture/known-gaps.md).
- **Editing is the shared modal ([ADR-0011](../../../docs/adr/0011-reusable-transaction-editing-modal.md)).** Each row opens the transaction-editing modal from an explicit Edit control; this module serves the editor content (the classification/category + transfer-destination controls) from an edit endpoint into the `core/templates` modal shell. A single Save issues both writes in turn (re-categorize, then the transfer mark for an outflow Transfer — each still validated, [ADR-0011]) and emits `transaction-changed` ([ADR-0010](../../../docs/adr/0010-event-driven-cross-region-refresh.md)); the edit endpoint is **view-agnostic** — it announces the change, it does not render the caller. Each list region self-refreshes on the event by re-fetching itself: the needs-attention worklist re-queries (a resolved row drops, the worklist shrinks), the default view shows the edited row in its new state. The home drill/wrap reuse the same editor and subscribe their own regions, so this module never learns about them.
- **The editor surfaces governing Rules ([ADR-0016](../../../docs/adr/0016-rule-editor-modal-and-cross-modal-return.md)).** It calls `categorization.RulesMatching` (the only new use of the existing `transactions → categorization` edge) to list the Rules matching the row's merchant (winner marked) and, when none match, offers a create-Rule control prefilled from the transaction. Those controls open categorization's rule editor modal **by URL only** (no view import) and pass this transaction's own edit endpoint as the opaque return handle, so saving, deleting, or dismissing the rule modal re-mounts this editor refreshed. categorization stays unaware of `transactions` — the edge runs one way as enforced by `architecture/isolation_test.go`.
- The recurring sync lives in `task.go` (implements `core/task.Task`) and is registered at the composition root (`server/services.go`).
