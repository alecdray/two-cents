# home — composing module

Rules: ../../../docs/architecture/archetypes/domain-module.md (domain-module-shaped, but owns **no tables and no repo**).

The dashboard's composition root: it turns the already-built primitives into the rendered read-side. Domain authority: [`docs/domain/README.md`](../../../docs/domain/README.md) §"Derived projections".

Module-specific notes:
- **It is the only module that imports multiple domain services** — `budget` + `transactions` + `categorization` + `accounts` — plus the pure `tracker` / `reporting` projections, `core/timex`, and `cfg.AppTimezone`. This is the legitimate dashboard composition root; the isolation test enforces that **nothing imports `home` except `server`**, and that `home` imports **no provider client** (it reaches the bank only transitively through the services it composes).
- **Owns no persistence.** No `repo.go`, no sqlc, no tables. It fetches through the services, fills the pure projections' local input structs, and joins Category **names** back onto the results — money is converted to dollars at the view boundary.
- **Pages:** `GET /{$}` the current-month Tracker (the landing page), `GET /wraps/{ym}` (`ym` = `YYYY-MM`) a single month wrap (the same GET also serves the wrap figure region's `transaction-changed` self-refresh), and the drill-down `GET /wraps/{ym}/spend/{bucket}` (likewise serves its region's self-refresh). The accounts overview moved to `/accounts`; `home` owns `/`.
- **Month-navigable home ([ADR-0018](../../../docs/adr/0018-month-navigable-home.md)):** the Tracker and the per-month wraps are one surface navigated by a horizontally-scrollable **month rail** — a chip per month from the earliest transaction's month through the current (newest, no future), chronological with the current month at the right; the active chip is the viewed month. Clicking a chip loads that month's page — `/` (Tracker) for the current month, `/wraps/{ym}` for earlier. The current month's `GET /wraps/{ym}` **redirects to `/`** (single canonical face; current-month drills return to the Tracker). There is no standalone wraps list. The composing layer builds the rail span for both pages.
- **Surplus** ([glossary](../../../docs/domain/README.md)): a closed-month figure (net income − savings contributed) shown on the **wrap only**, computed in `reporting`. The forward-looking Tracker carries none.
- **Tracker layout:** two tiers — top metrics (income + savings progress toward target; each drills into its underlying legs, [ADR-0012](../../../docs/adr/0012-wrap-income-savings-and-month-list-drill-ins.md)) over a **Budget** section of uniform budget rows (each Category, then everything-else), with **Total remaining** (their sum) as a gap-separated summary row below — then the inline **All transactions** list (the current month's whole set, the wrap's shared list component). The tiers + list sit in one inner **self-refreshing region** that re-serves on `transaction-changed` ([ADR-0010](../../../docs/adr/0010-event-driven-cross-region-refresh.md), mirroring the wrap's figure region); the month rail + label stay outside it. `CurrentMonthTracker` reads the month once through `transactions.MonthTransactions` (the joined read the wrap uses) and aggregates the figures from those same `RecentTransaction` rows, so figures and list reconcile from one set — there is no separate `ActivityRow` read path.
- **Drill-down ([ADR-0009](../../../docs/adr/0009-category-spend-drill-down.md), [ADR-0012](../../../docs/adr/0012-wrap-income-savings-and-month-list-drill-ins.md)):** `SpendDrill` lists the month's rows the `{bucket}` selects and sums them into the reconciling total, computed from the same rows it lists: signed-net Spending for a Category id / `uncategorized` / the `everything-else` residual; the Income legs for `income` (→ gross income); the savings-contribution source legs for `savings` (→ savings contributed). The residual reads the active-limit set from `budget.GetBudget` (which already drops archived limits, matching the Tracker's budgeted set) and is current-month only (`ErrResidualBucketUnavailable` otherwise) — the actuals-only `reporting` projection never reads a budget, so this budget-coupled bucketing lives here; `income`/`savings` read no budget and carry no month restriction. Rows are edited through the shared transaction-editing modal ([ADR-0011](../../../docs/adr/0011-reusable-transaction-editing-modal.md)); the drill region **self-refreshes** on the `transaction-changed` event ([ADR-0010](../../../docs/adr/0010-event-driven-cross-region-refresh.md)) — it re-queries and re-sums, so the total stays reconciled even when an edit moves a row out of the bucket. The region owns the refresh; the editor never learns about the drill.
- **Wrap figures + inline list ([ADR-0012](../../../docs/adr/0012-wrap-income-savings-and-month-list-drill-ins.md)):** the wrap surfaces **Net income** (income − spending, a derived summary, not itself a drill), gross **Income**, **Savings contributed**, and spend-by-Category; the Income/Savings/Category figures drill in. An inline editable list of the month's whole transaction set (every classification) sits under spend-by-Category; its rows open the shared modal, and because an edit there can shift any figure, the **whole wrap figure region** self-refreshes on `transaction-changed` (the figure region, like the drill region, is re-served by the wrap GET).
- **Time basis (ADR-0004):** the app timezone decides which calendar month "now" is and how many days remain (`timex.CurrentMonth` / `DaysLeftInclusive`); the month-range filter boundaries are UTC midnight (`timex.MonthRange`). **Transaction-date months are read from the stored calendar date directly (UTC), never re-zoned** — re-zoning a 1st-of-month UTC-midnight row would mis-bucket it (audit M1). The partial-edge flag reads the earliest transaction's month the same way (directly, UTC).
- **Signed cents (audit M2):** net spend is summed signed (`cents()` preserves sign, refunds reduce spend); income legs (inflows, negative) are negated to a positive total; savings is the sum of `savings_contribution` source legs. The no-budget fallback uses `budget.IsNoBudget`.
- **Partial flag:** a month is partial when it is at or before the earliest **transaction's** month — the backfill edge. It keys on the earliest transaction, not the connection's `created_at`: the provider backfills history from *before* the connect date, so anchoring to `created_at` would flag every backfilled month. With no transactions nothing is partial. **Caveat:** a single global edge — with multiple connections backfilled to different depths it can **under-flag** a shallower connection's own edge months. A precise per-connection history window is deferred (see `docs/roadmap.md`).
