# categorization — domain module

Rules: ../../../docs/architecture/archetypes/domain-module.md

Owns the classification taxonomy (built-in + custom Categories) and the user Rules, plus the pure decision engine. Domain authority: [`docs/domain/README.md`](../../../docs/domain/README.md) §Categorization.

Module-specific notes:
- **Decides, never writes transactions.** `ResolveCategorization` is a *pure* function (`categorization.go`) — it returns a `Decision` and writes nothing. Transactions remains the only writer of a transaction's classification/category. The engine takes all of its inputs (cleaned merchant, the Rule set, the taxonomy, the bank `Category`, the amount sign) — no storage access.
- **Precedence (first match wins):** manual override (defensive param) > Rule (longest matching substring; tie → most-recently-edited; archived-target rules skipped) > bank `personal_finance_category` primary (transfer signal → Transfer; INCOME → Income; else PFC→Category map → Spending + that built-in Category, archived target skipped) > direction fallback (outflow → uncategorized Spending; inflow → needs-review, never auto-Income). A spending-mapped inflow is a refund — negative Spending — never re-routed to Income.
- **Transfer subtype pairing (`ResolveTransferSubtype`, ADR-0003 layer 2).** Also pure (`categorization.go`): given an outflow Transfer leg and candidate inflow legs on *other* connected accounts, it pairs by exact-cent amount within a ±3-day window and reports the destination + subtype (a counts-as-savings destination ⇒ Savings contribution). The orchestration that loads the candidate set and re-resolves each sync lives in `transactions`.
- The PFC-primary → built-in-Category-id table is static in-code (`pfcSpendingCategory`); the twelve built-in Categories are seeded by the migration with the **stable string ids** the table references (`food_and_drink`, …). Keep the consts, the seed, and the map in lockstep.
- **Provider- and transaction-agnostic.** Imports `core/*` and `banking` (for `banking.Category`) only — never `transactions`, never `plaid`. Enforced by `architecture/isolation_test.go` (`TestCategorizationDependencyDirection`).
- The one cross-domain write — re-categorize matching transactions when a Rule changes — goes through the injected `ReapplyCategorization` func seam (mirrors accounts' `BackfillTransactions`). The composition root wires it; the `Service` calls it **after** a Rule mutation commits, with the affected substring(s) (old∪new on edit), and surfaces the returned count. **Category** create/rename/archive/unarchive never call the seam (archive keeps existing assignments; rename is id-stable; create has no existing matches).
- `repo.go` is the only file that touches `core/db/sqlc`; its methods take/return this package's domain types (`Category`, `Rule`), never `sqlc.*`.
- Archive-not-delete: a Category is never hard-deleted; archiving drops it from pickers and future auto-assignment only. Rules are hard-deleted.
- Validation failures surface as `ValidationError` (blank/duplicate category name; blank substring; a spending Rule with no Category) — adapters render the message inline (no redirect); other errors are 500s.
- The `Service.Resolve(ctx, bankCategory, merchant, counterparty, amount)` method is the published consumption point: it loads the active Rules + the taxonomy, cleans the merchant, and runs `ResolveCategorization`. Transactions calls it on every sync and from the rule-change re-categorizer; the module still writes no transaction rows.
- The server wires `ReapplyCategorization` to `Transactions.ApplyCategorization` (a late-bound closure at the composition root), so a Rule create/edit/delete re-categorizes matching non-overridden transactions and the "N re-categorized" count is real. The transactions-side auto-categorize-on-sync and the re-categorize picker on `/transactions` consume this module through the `transactions→categorization` import edge (enforced by `architecture/isolation_test.go`).
