# App-wide request feedback

Two layers of in-flight feedback for an HTMX UI that previously gave none:

1. An **app-wide pending indicator** — a thin progress bar fixed to the top of the viewport — driven by the **aggregate HTMX request lifecycle**: visible while at least one request is in flight, hidden once they settle. Because it reads the request lifecycle rather than any one element, a single instance in the shared layout covers every interaction — boosted navigation, form posts, fragment and modal loads — with no per-element wiring.
2. **Action-result feedback for "Sync now"** — an in-progress state on the control while its request runs, and a brief **transient inline confirmation** on success.

The interaction model is server-rendered HTML over HTMX ([principles](../design/principles.md)), and until now nothing signalled that a request was in flight; any non-instant response read as a frozen app. The forcing case is the manual sync: the first sync after a cursor-clearing migration is a slow full re-pull, and because the newer bank detail is modal-only the list re-renders **identically** — so a successful sync looked like nothing happened. The two layers answer two different questions: the bar says *something is in flight*; the confirmation says *that action finished, and here is its result*.

A few consequences shape the decision:

- **The indicator is app-wide chrome, mounted once in the shared layout** — alongside the other always-present elements — not wired onto interactive elements. It is driven by the request lifecycle as **client-only ephemeral state**, with no server round-trip and no server-rendered state of its own, consistent with the [principle that scripting is reserved for genuinely client-only state](../design/principles.md). Per-element indicators would have to be attached to every interactive element and would still miss boosted navigation; one lifecycle-driven instance covers all of it.
- **It tracks the aggregate lifecycle and must clear on settle regardless of outcome** — success, server or network error, or an aborted/superseded request. A stuck bar is worse than no bar, so "idle ⇒ hidden" is the load-bearing invariant, not "hide on success."
- **It coexists with existing chrome without disturbing it** — the fixed bottom navigation ([ADR-0014](0014-bottom-bar-navigation.md)), the modal container, and the event-driven / OOB self-refreshing regions ([ADR-0010](0010-event-driven-cross-region-refresh.md)). It sits at the top edge above content and owns no domain state.
- **Pending and result are separate concerns.** The bar is purely *pending* feedback; a completed action that changes nothing visible still needs a *result* acknowledgement. So the sync action carries its own confirmation, rendered by the acting handler into the region it already swaps — not an OOB swap, not an event fan-out ([ADR-0010](0010-event-driven-cross-region-refresh.md)), not a shared overlay — in the same inline slot where a recoverable sync error already appears, so the result lands where the action was taken. The confirmation is **transient**: it auto-clears so it never lingers on an idle page.
- **Distinct from the app's other HTMX patterns.** Unlike the reusable modal ([ADR-0011](0011-reusable-transaction-editing-modal.md)) and event-driven refresh ([ADR-0010](0010-event-driven-cross-region-refresh.md)), the bar fetches and renders no domain content, and the sync confirmation is single-region, single-handler — neither introduces a cross-region or cross-module coupling.

Rejected: **per-element HTMX indicators** (the built-in spinner attached to each control) — they must be wired onto every interactive element and a boosted navigation still needs a global treatment, where one lifecycle-driven bar covers everything once; **a general transient toast/notification primitive** for the confirmation — more machinery than a single action warrants, where the existing inline error slot is the established place to surface a sync's outcome (a shared notification system, if ever warranted, is its own decision); **surfacing an updated count or last-synced time** in the confirmation — deferred, as it needs a sync summary the service does not return today, and a plain transient acknowledgement already defeats the "looks like nothing happened" problem.

The indicator's markup, animation, and which lifecycle events it binds, and the sync confirmation's exact rendering, are presentation details that live with the owning primitive and templ and the [design-system](../design/design-system.md) vocabulary — reconciled to the shipped code, not fixed by this decision.
