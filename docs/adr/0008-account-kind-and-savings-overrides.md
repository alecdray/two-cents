# Account kind and counts-as-savings overrides

[ADR-0005](0005-spending-tool-three-bucket-account-kind.md) established that an Account's **kind** (`cash` / `credit` / `other`) is seeded from the bank's reported type but **user-overridable**, and the domain model has long carried the operations to override kind and to set the **counts-as-savings** flag, each guarded so a sync never reseeds a user's choice. This decision records how those overrides are finally surfaced — an inline picker on each row of the accounts overview — and the two non-obvious behaviours that surfacing them forces.

**counts-as-savings is offered on every bucket except `credit`.** The flag is *orthogonal* to kind, and the transfer-subtype pairing ([ADR-0003](0003-two-layer-transfer-detection.md) layer 2) honours that: it reads only the flag, never the destination's kind. Saving into a retirement or brokerage account — an `other`-kind Account, held out of net cash — is still saving, so the toggle belongs there as much as on a checking or savings account. The one kind where it is meaningless is `credit`: a Transfer into a credit Account is by definition a **Credit-card payment**, a plain Transfer, never a Savings contribution. So the picker offers the savings toggle on `cash` and `other` rows and withholds it on `credit`.

**Overriding an Account to `credit` force-clears its counts-as-savings flag — a deliberate coupling at that one boundary.** This is the surprising part, because it contradicts the orthogonality otherwise asserted. The reason is that the pairing engine *trusts* the flag with no kind check — a credit destination is expected to simply carry the flag false and fall out as plain. A reachable path breaks that trust: a bank-type savings Account seeds to `cash` with the flag on, then the user overrides its kind to `credit`. Were the flag left set, Transfers into a credit card would resolve as Savings contributions, double-counting a debt payment as saving. We chose to preserve the engine's invariant by clearing the flag at the source — as part of the kind override when the target is `credit` — rather than hardening the pure engine with a kind-aware branch. The engine stays a single-discriminator function, and the coupling lives in exactly one place. `cash`↔`other` overrides never touch the flag, so orthogonality holds everywhere it is meaningful.

**An effective change to counts-as-savings eagerly re-pairs existing Transfers.** Kind overrides have immediate visible effect on the overview — net cash recomputes and the row re-buckets in the swap. A counts-as-savings change has *no* effect on that page; it only changes which Transfers are Savings contributions, read downstream by the Tracker and Reporting. Left to the next scheduled sync, the toggle would appear to do nothing for hours. So a flag change — the direct toggle in either direction, or the implicit clear from a `credit` override — triggers an immediate re-pairing of the stored Transfer legs (no provider call), through the same injected-seam pattern used when a Rule change re-categorizes or a connect backfills, which keeps the accounts domain free of a direct dependency on transactions. The re-pair skips manually-overridden Transfers, so a user's sticky transfer-destination choice is never undone by a flag flip. A pure `cash`↔`other` kind change leaves the flag untouched and does **not** re-pair.

Consequences. The accounts domain gains a third injected post-action seam, and the transactions domain exposes its existing stored-leg re-pairing as a callable operation. The overview row grows a kind picker and a conditional savings toggle, both applied in place and re-rendering the row's own controls (the savings toggle appearing or vanishing as the kind crosses the `credit` boundary). Hidden and closed Accounts are not shown on the overview, so there is no override path for them — consistent with the page listing only active Accounts. Reverting an override to the bank-seeded default is intentionally out of scope: a user picks the bucket they want, and the seed simply stops applying.

Rejected: showing the savings toggle on `cash` rows only — hides it from `other` Accounts where saving into retirement/brokerage is exactly the case it exists for, and strands a seeded-on flag with no way to clear it after an override to `other`. Rejected: hardening the pairing engine with a "credit destination is never savings" check instead of clearing the flag — spreads the credit special case into the pure engine and leaves a contradictory stored flag on the Account. Rejected: go-forward-only re-pairing (let the next sync catch up) — makes the savings toggle feel inert, with the Tracker stale for hours after an explicit user action. Rejected: a "bank default" reset choice in the picker — a re-seed path and its own surface for a capability no one asked for in v1.
