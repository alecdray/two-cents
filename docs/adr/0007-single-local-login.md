# Single local login

Two Cents is a single-user, self-hosted service ([ADR-0001](0001-self-hosted-single-user-service.md)), but every page was open to anyone who could reach the port. This decision puts the whole application behind **one local login**: a single credential the operator controls, verified server-side, with an authenticated session required for every page except the login screen and static assets. There is no third-party identity provider — the deviation from `wax`'s Spotify OAuth that [ADR-0001](0001-self-hosted-single-user-service.md) already anticipated.

The login is **password-only**. Because there is exactly one user by design, a username challenge is friction with no security value; the operator proves they are the operator by knowing the password. The credential is stored as a hashed password (never the plaintext, never reversible) in its own table, alongside a stable identifier, rather than in an environment variable: the database already lives on the mounted volume that holds the encrypted bank access tokens, so it is the natural home for the one other secret-bearing record, and a stored row can be rewritten to change the password later. The single-user rule is a **design** constraint, not a schema-enforced one — the table can technically hold more than one row, but the product only ever seeds and uses one.

The session is a signed token in an `HttpOnly` cookie, re-issued on each request so active use never expires while a long idle gap does (a multi-week sliding window). The sensitive secrets — the per-Item bank access tokens — never enter the token; they stay encrypted server-side, so the cookie's blast radius is "can view my finances," not "can move my money." That is why a generous session lifetime is an acceptable trade for not re-authenticating a personal tool on every visit.

The session **carries the real stored user identifier into request context**, even though no domain data is partitioned by user. Nothing is filtered by that id — there is one user — so authorization is **binary**: a valid session reaches everything, an invalid one reaches only the login screen. Carrying a genuine identifier rather than a constant keeps the request-context plumbing honest and leaves a seam should "single-user" ever be revisited.

The credential is **created and rotated out-of-band**, through a command the operator runs on the box — the same operational model as applying migrations — not through an in-app account screen. With no credential configured, the login screen says so plainly rather than offering an unauthenticated way to set one. In-app password management is deliberately out of scope for v1.

Consequences. End-to-end tests, which previously navigated straight to pages because nothing gated them, now authenticate once through the real login and share that session across the suite — the gate is exercised rather than bypassed, while per-test setup still seeds its own data directly. A future reader should note what is intentionally absent for v1: login rate-limiting or lockout, CSRF defence beyond the cookie's same-site restriction, and any second user or role. Each is reasonable to add later but was judged unnecessary for a single-operator, self-hosted box.

Rejected: a hashed password in an environment variable — no path to in-app rotation and a stranger operational story than a record sitting beside the other at-rest secret. Rejected: a username + password challenge — standard, but pure ceremony for a single user. Rejected: an in-app first-run wizard to set the password — an unauthenticated write endpoint and a whole flow, for a box the operator already configures from the command line. Rejected: disabling the gate in local or test builds to keep tests simple — that leaves the gate untested and risks shipping it effectively off.
