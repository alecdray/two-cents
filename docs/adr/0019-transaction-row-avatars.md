# Transaction-row avatars — merchant logo over category glyph

Every transaction row gains a leading avatar for faster visual scanning: the merchant's logo when we hold it, otherwise a **category-colored glyph** keyed to our own taxonomy. The avatar renders from **local state only** — the merchant logo is proxied and cached on our origin, never hotlinked — so a single-user, self-hosted app never leaks which merchants a user transacts with, and the request path carries no third-party dependency.

Rows were text-only, and the two obvious icon sources pull in opposite directions. Merchant logos ([0013](0013-richer-bank-transaction-detail.md) already ingests `logo_url` + the merchant entity id) give recognition but exist for only some merchants and live on a remote CDN. A category glyph is always available and consistent but generic. So we use both, ranked: **logo when cached, glyph otherwise** — full coverage with recognition where it exists.

A few consequences shape the decision:

- **The glyph set is ours, keyed to the taxonomy — not the bank's category icon.** [0013](0013-richer-bank-transaction-detail.md) deliberately rejected Plaid's `personal_finance_category_icon_url`; that stands. The glyph and color are a static in-code mapping over the built-in [Category](../domain/README.md) ids (the same stable ids the PFC→Category table already keys on), with a deterministic color for custom categories and fixed defaults for the category-less classifications (income, transfer, uncategorized). Nothing is added to the `Category` entity: the glyph and color are derived, not stored, so there is no migration and no user-pickable icon to maintain. (The logo cache below is a separate, rebuildable store.)

- **Category colors are a new token group, not the semantic palette.** The theme's semantic tokens (success, error, …) are few and meaning-loaded; pressing them into service as arbitrary category hues would break "reach for the role, not the color." A distinct **categorical palette** is added to the theme and documented as design tokens ([design-system](../design/design-system.md)) — semantic names, defined once in the stylesheet, referenced by the glyph and its tint.

- **Logos are proxied and cached on our origin, never hotlinked.** Hotlinking `logo_url` from the browser would leak the user's transaction history (which merchants, when) to a third-party CDN on every page view, break behind the private network, and rot with the link. Instead the logo bytes are fetched once server-side and cached locally, keyed by a **content hash of the logo URL**, and the row references our own image endpoint. Keying by the URL hash makes the cache self-invalidating: the logo URL is a bank-refreshed display-detail field, so when the bank changes it the new URL is simply a new key that warms afresh, with no invalidation logic and no stale serve. A **negative-cache** entry records a URL that yields no usable logo (absent, failed, or refused by the fetch policy below) so the warm step attempts each key exactly once instead of retrying a dead fetch every sync; a merchant that later gains a logo arrives under a new URL/key and so is retried automatically.

- **Fetching is a best-effort step after sync; rendering reads only the cache.** The cache is warmed as a discrete step after the transaction sync upsert — sync already does network I/O and walks the rows — and its failures are isolated from sync correctness, the same decoupling the categorization sweep already takes. The step considers the whole stored set (not just the pull's delta, so pre-existing merchants heal without a cursor-clear) but fetches only a **bounded batch of un-cached merchants per pass, most-recent-first**, so the backlog drains over successive syncs rather than in one fetch storm and the merchants on the current month warm first. Because render consults only the local cache, a cache miss simply shows the glyph; a merchant's logo appears once its warm pass reaches it. No client-side error-swap, no image-box flicker, no remote call on the request path.

- **Fetching stored URLs is an SSRF surface, so it runs behind a constrained seam.** The URLs originate from bank data, not the user, but are attacker-influenceable in principle. `transactions` fetches through an injected `LogoFetcher` interface it defines — provider-agnostic and faked in tests, so no unit test touches the network (mirroring the `BankProvider` / `fakebank` seam). The concrete fetcher, wired at the composition root, is the provider adapter's concern and holds an **in-code host allowlist** (no new config — the host is a fact of that adapter, not a deployment knob); it requires `https`, refuses any host outside the allowlist, bounds the response by time and size, and accepts only **raster image** content types — SVG is excluded so a script-bearing image is never fetched or served. Anything failing a guard is treated as no-logo (negative-cached, glyph shown). The served bytes carry their stored content type, `nosniff`, and an immutable cache header (safe — the key is a content hash).

The avatar is one row element shared by every transaction-row surface (the transactions list, the wrap and Tracker month lists, and the spend drill), so it lands everywhere at once. Which glyph maps to which category, the exact palette, and the avatar's size and shape are presentation details owned by the modules and the stylesheet, not part of this decision.
