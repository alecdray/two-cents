# Bank provider selected by configuration

The bank provider the app wires in is chosen at the composition root by a single configuration value. The default selects the live provider that reaches the real bank network; one other value selects a deterministic in-process stand-in that returns a fixed set of accounts and balances and no transactions, with no network and no provider credentials. Everything downstream still depends only on the `BankProvider` seam ([ADR-0002](0002-bankprovider-abstraction.md)); only which concrete provider is constructed changes.

The fake is the same kind of in-memory `BankProvider` the project already relied on to unit-test consumers without the provider client. This decision **promotes that unit-test fake to a wiring option**: instead of existing only inside tests, it becomes a provider the running application can be configured to use. When it is selected, the real server runs unchanged — real SQLite, real HTTP, real templ rendering, the real connection flows — and only the leaf provider is swapped. So the connect, reconnect, and account-listing flows can be exercised end to end against the actual server with stable, assertable data, with **no browser-side mocking** and no live bank.

Why a wiring option rather than only a test double: the connection flows span the front end (the connect UI keyed off the link token's mode), the HTTP handlers, persistence, and the provider. A test double substituted inside a unit test exercises none of that integration; a hosted-provider sandbox requires credentials and live calls and yields non-deterministic data that end-to-end assertions can't pin to. Selecting the fake at the root closes both gaps: the whole stack runs, and the data is fixed, so an end-to-end test can drive the real UI and assert on exact accounts and balances. The selection lives at the composition root because that is the one place allowed to name concrete providers (manual DI); a small selection helper keeps the fake branch constructible — and unit-testable — without building the real client or its credentials.

Consequence: the real provider's credentials are required only when it is the one selected; the fake branch needs none, so a developer or an end-to-end run can start the app with nothing but the fake configured. The stand-in's accounts, balances, and tokens are deliberately fixed, because end-to-end tests assert on them; changing them is a change to those tests too. The fake remains a dependency-graph leaf fenced exactly like the real provider — only the provider client and the composition root may import it — so promoting it to a wiring option does not weaken the seam the abstraction depends on.

Rejected: exercising the flows only against the live provider's sandbox — needs credentials, makes network calls, and returns data too variable to assert on. Rejected: mocking the provider in the browser/front end — leaves the server's own handling of the flows untested and couples the tests to front-end internals, the opposite of what an end-to-end test should verify. Rejected: keeping the fake confined to unit tests — never exercises the integration across the front end, handlers, persistence, and provider that the connection flows actually span.
