# Architecture Decision Records

Short entries that capture **why** a decision was made, when the rationale would otherwise be lost once the old approach is gone.

## Decision log

| # | Decision | Summary |
|---|---|---|
| [0001](0001-self-hosted-single-user-service.md) | Self-hosted single-user service, mirroring wax | One Go + SQLite binary in a Docker container the user controls; stack and archetypes adopted wholesale from the sibling `wax` project, with single local login instead of third-party OAuth. |
| [0002](0002-bankprovider-abstraction.md) | Bank access behind a BankProvider abstraction | All bank data flows through a `BankProvider` interface returning our own domain types. We chose Teller, then switched to Plaid (Teller closed self-serve signup; Plaid's Trial plan unblocks signup and its `personal_finance_category` upgrades categorization) — the interface made it an adapter swap, not a rewrite. |
| [0003](0003-two-layer-transfer-detection.md) | Two-layer transfer detection | Transfers are detected first by the bank-provided category's primary level, then by pairing the inflow leg on another connected account — because Plaid exposes no destination-account reference. |
| [0004](0004-configured-app-timezone.md) | Single configured timezone for time reckoning | "Today," days-left-in-month, and the current month are reckoned in one configured app timezone (default EST), not a per-request browser zone — so scheduled jobs share the clock and month boundaries stay stable. |
| [0005](0005-spending-tool-three-bucket-account-kind.md) | Spending tool — three-bucket account kind | Two Cents is a spending tool, not a net-worth tracker. Account kind is `cash | credit | other` (depository / cards / everything else); loans, mortgage, and investments are tracked as `other` and excluded from net cash. |
| [0006](0006-bank-provider-selected-by-config.md) | Bank provider selected by configuration | The composition root picks the `BankProvider` by config (`BANK_PROVIDER`): the default reaches the live bank network, a "fake" value selects a deterministic in-process stand-in. Promotes the unit-test fake to a wiring option so the connection flows run end to end against the real server with no browser mocking and no live bank. |
| [0007](0007-single-local-login.md) | Single local login | The whole single-user app sits behind one password-only login; the credential is a hashed row in the DB (not config, rotated by a CLI command), the session is a sliding `HttpOnly` cookie, and no domain data is partitioned by user, so authorization is binary. |
| [0008](0008-account-kind-and-savings-overrides.md) | Account kind and counts-as-savings overrides | The kind/counts-as-savings overrides are surfaced as an inline per-row picker on the accounts overview. The savings toggle is offered on every bucket but `credit`; overriding an Account to `credit` force-clears its counts-as-savings flag (a deliberate coupling that preserves the pairing engine's no-kind-check invariant); and an effective flag change eagerly re-pairs existing Transfers. |
| [0009](0009-category-spend-drill-down.md) | Category spend drill-down | Drilling from a spend-by-Category figure into its transactions is a dedicated composer-owned read view nested under the wrap URL, not a filter on the activity list, so the list provably reconciles to the figure. One `{bucket}` selector covers a Category, uncategorized, and the budget residual; the residual is current-month only and bucketed in the composer (the actuals-only reporting projection never reads a budget); editable rows re-render the whole region to keep the net total honest. |
| [0010](0010-event-driven-cross-region-refresh.md) | Event-driven cross-region refresh | When an action must update regions beyond its target, the handler emits an HTMX event and each affected region re-fetches itself; OOB swaps are reserved for tightly-coupled siblings the handler already renders in the same template. Chosen so a reusable editor invoked across views stays context-free (it never imports or renders its callers' regions) — trading extra self-refresh requests, irrelevant on a single-user app, for low coupling. |
| [0011](0011-reusable-transaction-editing-modal.md) | Reusable transaction-editing modal | Transaction editing is one modal editor reused by every view that lists transactions — replacing the per-view inline pickers that otherwise drift. A domain-free modal shell (primitive) hosts transactions-owned editing content; a single Save composes the existing two writes (each keeping its own coupling validation, not a merged write) and emits the change event so callers self-refresh, keeping the editor context-free and shareable. |
| [0012](0012-wrap-income-savings-and-month-list-drill-ins.md) | Wrap income/savings/full-month drill-ins | The wrap's gross Income and Savings-contributed figures become drill `{bucket}`s reconciling to their underlying legs (income legs; savings-contribution source legs) — actuals-only, any month, reading no budget. Net income stays non-drillable (a derived net with no single underlying set), so a distinct gross Income figure is the drill entry. A full-month transaction list under spend-by-Category is edited through the shared modal, with the whole wrap figure region self-refreshing on the change event since an edit there can shift any figure. |
| [0013](0013-richer-bank-transaction-detail.md) | Richer bank transaction detail | A Transaction carries read-only bank display detail beyond the cleaned merchant — raw descriptor, merchant identity (logo/website/entity id), payment channel, categorization confidence, authorized/posted timestamps, and a structured counterparties list. It is display-only (never a categorization input, so the cleaned-merchant invariant holds) and bank-sourced, so it lives *in* the sync upsert — opposite the override facets that are excluded from it. Counterparties is a denormalized JSON column; existing rows backfill by clearing the sync cursor. Field set validated against real production data (`original_description` dropped as empty). |

| [0014](0014-bottom-bar-navigation.md) | Bottom-bar navigation | Primary navigation moves from a top strip listing every destination to a fixed bottom bar carrying a few primary destinations plus a "More" overflow — the thumb-reachable mobile idiom for a phone-first app, where the strip stopped fitting a narrow viewport as destinations grew. One layout at every width (the bar stays anchored flush to the bottom, centered to the content column, rather than floating inset on desktop or reverting to a top strip), the navbar gains the active destination as a checked value from its own destination set (staying domain-free, since the request path is not in context), and the overflow is client-only static links — distinct from the HTMX modal ([0011](0011-reusable-transaction-editing-modal.md)) and event-driven refresh ([0010](0010-event-driven-cross-region-refresh.md)). |
| [0015](0015-app-wide-request-feedback.md) | App-wide request feedback | An HTMX UI that gave no in-flight feedback gains two layers: a top-of-viewport progress bar driven by the aggregate request lifecycle (one instance in the shared layout covers boosted navigation, posts, and fragment/modal loads; client-only, and the load-bearing invariant is idle ⇒ hidden, clearing on success/error/abort), and action-result feedback for "Sync now" (in-progress control state plus a transient inline confirmation, since the bar signals *pending* but a sync that re-renders an identical list otherwise looks like nothing happened). Rejected per-element indicators, a shared toast primitive, and surfacing a count/last-synced time (deferred). |
| [0016](0016-rule-editor-modal-and-cross-modal-return.md) | Rule editor modal and cross-modal return | Creating/editing a Rule is one modal editor (the second consumer of the shared shell, [0011](0011-reusable-transaction-editing-modal.md)), opened from the Rules page and from the transaction editor — which now surfaces the Rules matching a transaction (winner marked) and a prefilled create when none match. Opening from another modal replaces it (the shell mounts one at a time), so the origin passes an opaque same-origin reopen handle the editor echoes back on success to re-mount the refreshed origin — keeping categorization ignorant of transactions. The cross-module read (which Rules match) reuses the precedence engine and imports no view; a Rule change announces the change event so transaction views self-refresh, while the Rules list — the handler's own sibling — is re-rendered in place ([0010](0010-event-driven-cross-region-refresh.md)). |
| [0017](0017-custom-account-names.md) | Custom account names | A user can rename any Account from the overview, overriding the bank name everywhere it shows. The override is a nullable `custom_name` column (not the boolean-flag pattern of [0008](0008-account-kind-and-savings-overrides.md)): the bank name stays in `name` and keeps refreshing on sync, and non-NULL `custom_name` is itself the override signal. `Account.DisplayName()` is the one precedence point; the overview rows and `ConnectedAccountFacets` switch to it, so the transactions picker and transfer-pairing inherit it for free. Empty input clears back to the bank name; a rename fires no transfer re-pair. |
| [0018](0018-month-navigable-home.md) | Month-navigable home | The current-month Tracker and the retrospective month wraps ([0012](0012-wrap-income-savings-and-month-list-drill-ins.md)) unify into one month-navigable surface: a scrollable rail of month chips spanning the earliest transaction's month to the current one selects the month, showing the budget-relative Tracker for the open month and each earlier month's actuals-only wrap. The current month is the newest reachable month (no future); reusing the per-month wrap address keeps the drill-downs and their self-refresh ([0009](0009-category-spend-drill-down.md), [0010](0010-event-driven-cross-region-refresh.md)) untouched. The standalone wraps list, the month-summary read behind it, and its overflow link ([0014](0014-bottom-bar-navigation.md)) are removed — the rail is the sole month navigator, mooting a richer wraps list. |
| [0019](0019-transaction-row-avatars.md) | Transaction-row avatars | Every transaction row gains a leading avatar — the merchant logo when cached, otherwise a category-colored glyph keyed to our own taxonomy (not Plaid's category icon, still rejected per [0013](0013-richer-bank-transaction-detail.md)). Glyph + color are a static in-code mapping over the built-in Category ids with a deterministic custom-category color and classification defaults, so nothing is stored or migrated. Category colors are a new categorical token group, distinct from the meaning-loaded semantic palette. Logos are proxied and cached on our origin (never hotlinked — no per-view leak to a CDN on a self-hosted app), warmed by a best-effort step after sync and rendered from local cache only; the server-side fetch is constrained to `https` on the provider's logo CDN to bound the SSRF surface. One shared avatar element, so it lands on every transaction-row surface at once. |

## Format

Lead with the decision — 1–2 sentences naming what was decided — under an `# h1` title. Then add only the **minimal context needed to understand that decision and its implications**: the constraint or trade-off that forced it. Rejected alternatives and consequences earn space only when they carry real weight, and as a brief clause or short list — never an obligatory section. An ADR that runs past a few short paragraphs is usually restating things that belong elsewhere.

The **current** state is the codebase — don't restate it. Keep implementation details out: no file names, class names, function names, or exact UI strings that a routine refactor would invalidate. If a sentence would need to change after such a refactor, it doesn't belong here.

## Naming

`NNNN-short-slug.md` — four-digit zero-padded prefix, never renumbered. A decision that replaces an earlier one is a new ADR; reference the prior number in the body. Don't edit the old one.

## When to write one

Write an ADR when a change replaces a meaningful prior approach, or locks in a foundational choice, that a future reader would otherwise wonder *"why is it like this?"* about. Routine changes don't qualify.
