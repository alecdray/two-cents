# Rule editor modal and cross-modal return

Creating and editing a Rule is **one modal editor**, the second consumer of the shared modal shell ([ADR-0011](0011-reusable-transaction-editing-modal.md)) — replacing the Rules page's inline create form and per-row edit form. It opens from the Rules page and from the transaction editor, and when it opens from another module's modal it **returns to that origin**: the opening surface hands the editor an opaque return handle — a same-origin reopen URL — which the editor echoes back — on a successful save or delete, or on dismissal — to re-mount the origin, refreshed.

The reason it must be one modal reachable from two places is the feature: the transaction editor now surfaces the Rules that match a transaction's merchant (the winner of the precedence engine marked), so a user can jump from a transaction to the Rule governing it, or — when none match — create one prefilled from that transaction. Both jumps want the *same* create/edit experience the Rules page offers, and both want to land the user **back at the transaction** they came from once the Rule is saved and the transaction re-categorized.

The constraint is that only one modal mounts at a time (the shared shell swaps a single fixed container, [ADR-0011](0011-reusable-transaction-editing-modal.md)), so opening the rule editor *replaces* the transaction editor. Returning to it must not couple the two modules:

- **The origin is an opaque handle, not a known destination.** The transaction surface passes the reopen URL; the rule editor treats it as a same-origin path to re-mount on save or dismiss, and never constructs it (a handle carrying a scheme or host is rejected, never followed). So the categorization module stays ignorant of transactions — it returns to whatever handed it the handle. The Rules page passes no handle, so its save closes the modal and the handler refreshes the list in place.
- **The cross-module read carries no view dependency.** The transaction editor asks categorization which Rules match a merchant (a direction already open, transactions → categorization) and renders controls that open categorization's modal by URL — it imports no categorization view, and categorization still never imports transactions. The match set is decided by the same precedence engine that categorizes, so "which Rule applies" and "which Rule actually applied" cannot drift.
- **A Rule change reaches other views by event; its own list by direct render.** A save or delete announces the change event ([ADR-0010](0010-event-driven-cross-region-refresh.md)) so the *transaction* views a re-categorization touches self-refresh — the editor never renders them, staying context-free exactly as the transaction editor is. The Rules list, by contrast, is the sibling the categorization handler already produces, so it is refreshed in the same response (its region re-rendered, the modal closed out-of-band) — the OOB-for-tightly-coupled-siblings half of ADR-0010, not a separate event.

Rejected: stacking the two modals (the shell mounts one at a time by design, and nesting it to make the origin survive would complicate a primitive every page loads, to recover context a reopen already restores); closing on save without returning (loses the transaction the user came from); letting the transaction surface import the rule editor or hard-code categorization's reopen of it (the coupling the opaque handle exists to avoid). Prefilling the new Rule from the transaction's current outcome, and listing every matching Rule rather than only the winner, are presentation choices owned by the modules and their templ, not part of this decision.
